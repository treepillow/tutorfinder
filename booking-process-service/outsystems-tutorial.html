Part 1 — Setup

Get OutSystems running and create your application

1

Get OutSystems Access

  1. Go to outsystems.com/free and sign up for a free Personal Environment
  2. Download and install Service Studio from the developer portal
  3. Open Service Studio → click Connect → enter your environment URL
    https://yourname.outsystemscloud.com
  4. Sign in with your OutSystems account credentials
2

Create the Application

  1. In Service Studio, click New Application
  2. Choose Service (not Reactive Web — this is backend-only)
  3. Name it: TutorFinder_BookingProcess
  4. Click Create App → then Create Module
  5. Name the module: BookingProcessLogic
Why Service type? A Service module exposes REST APIs and runs Timers without a UI layer. It's the correct type for a backend orchestrator.

  2. Right-click REST → Consume REST API

  4. Fill in the method name, HTTP verb, and URL from the tables below

  6. Click Finish — repeat for each method

Part 4 — Server Actions (Business Logic)

Build each orchestration flow in the Logic tab

How to create a Server Action Logic tab → right-click Server Actions → Add Server Action → name it → add Input/Output Parameters in the right panel → drag flow nodes from the toolbar on the left.
5

InitiateBooking — Tutee selects a slot

Input Parameters

TuteeId Integer   TutorId Integer   AvailabilityId Integer   LessonDate Text   StartTime Text   EndTime Text   Amount Text

Output Parameters

BookingProcessId Integer   BookingId Integer   Success Boolean   ErrorMessage Text

Flow

Start │ ▼ [Try] ─────────────────────────────────────────────────── [Catch] │ │ ▼ ▼ UpdateSlot (AvailabilityService) [Assign] availability_id = AvailabilityId Success = False status = "Reserved" ErrorMessage = AllExceptions.Message │ ▼ CreateBooking (BookingService) tutor_id, tutee_id, availability_id, lesson_date, start_time, end_time │ ▼ GetProfileInternal (ProfileService) user_id = TutorId │ ▼ SendNotification (NotificationService) user_id = TutorId type = "Booking" phone_number = GetProfileInternal.phone message = "New booking request on " + LessonDate + " at " + StartTime + ". Please confirm or reject in the app." │ ▼ [CreateBookingProcess] ← drag from Data tab BookingId = CreateBooking.booking_id TuteeId = TuteeId TutorId = TutorId AvailabilityId = AvailabilityId Amount = Amount ProcessStatus = "AwaitingConfirmation" │ ▼ [Assign] BookingProcessId = CreateBookingProcess.Id BookingId = CreateBooking.booking_id Success = True │ ▼ End

Part 6 — Timers (Scheduled Tasks)

Automatically cancel bookings that exceed the 24-hour timeout

12

Timer: ExpireUnconfirmedBookings

  1. Click the Processes tab → right-click Timers → Add Timer
  2. Name it ExpireUnconfirmedBookings
  3. Set Schedule to 0 * * * * (every hour)
  4. Under Action, create a new Server Action: HandleExpiredConfirmations

Flow inside HandleExpiredConfirmations

GetExpiredBookings (BookingService) type = "confirmation" │ ▼ [For Each] booking in GetExpiredBookings.bookings │ ├── CancelBooking (BookingService) │ booking_id = booking.booking_id │ ├── [GetBookingProcessByBookingId] ← Aggregate │ ├── UpdateSlot (AvailabilityService) │ status = "Available" │ ├── GetProfileInternal → tutee phone │ ├── SendNotification │ message = "Booking cancelled: tutor did not confirm within 24 hours." │ └── [UpdateBookingProcess] ProcessStatus = "Cancelled"
13

Timer: ExpireUnpaidBookings

Same setup as Step 12 with these differences:

  • Timer name: ExpireUnpaidBookings
  • Action name: HandleExpiredPayments
  • Call GetExpiredBookings with type = "payment"
  • Notification message: "Booking cancelled: payment not received within 24 hours."
Timer Retries OutSystems automatically retries failed Timers on the next scheduled run — you don't need to write retry logic yourself.

Update Site Properties (Base URLs)

OutSystems runs outside your Docker network, so container hostnames won't resolve. Replace them with your machine's LAN IP.

  1. Go to Service Center: https://yourname.outsystemscloud.com/ServiceCenter
  2. Factory → Modules → BookingProcessLogic → Site Properties
  3. Update each URL to use your machine's IP (e.g. 192.168.1.x) instead of service names
PropertyLocal Docker Value
AvailabilityServiceBaseURLhttp://192.168.1.x:5003
BookingServiceBaseURLhttp://192.168.1.x:5004
PaymentServiceBaseURLhttp://192.168.1.x:8080
NotificationServiceBaseURLhttp://192.168.1.x:5007
ProfileServiceBaseURLhttp://192.168.1.x:5001
Find your LAN IP Run ipconfig getifaddr en0 in Terminal (Mac) to find your local IP address.

Common Mistakes to Avoid

MistakeFix
Not handling REST errorsWrap every REST call in a Try/Catch block — unhandled exceptions will crash the entire flow
Notification failure breaking the bookingPut SendNotification calls in their own separate Try block so SMS failure doesn't roll back the booking
Timer not runningCheck Service Center → Monitoring → Timers — make sure the timer shows as Active
JSON field names wrongOutSystems uses PascalCase by default but your APIs use snake_case — check the Name field in each REST parameter and change it to snake_case
Can't reach Docker containersUse your machine's LAN IP (not localhost or container names) in Site Properties — OutSystems cloud can't resolve Docker internal DNS
Aggregate returns no rowsAdd a null check after every Aggregate — if Count = 0, raise an exception with a clear message
Long Integer vs Integer mismatchPaymentId is Long Integer in OutSystems but the REST API returns a number — make sure the types match or you'll get a conversion error
